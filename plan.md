# VRC Fish Fishing Bot - 實作計畫

## 整體架構

```
main.py                 - 主程式入口，狀態機迴圈
config.py               - 設定檔（ROI 區域、閾值、延遲等）
screen_capture.py       - 畫面擷取模組
state_detector.py       - 各階段視覺判斷
audio_detector.py       - 咬勾提示音偵測
tension_handler.py      - 拉力計階段控制邏輯
input_handler.py        - 滑鼠/鍵盤模擬輸入
utils.py                - 工具函式（debug overlay、logging）
```

依賴套件：`mss`（快速截圖）、`opencv-python`、`numpy`、`pyaudio` 或 `sounddevice`、`pywin32` 或 `pyautogui`

---

## 狀態機設計

```
IDLE ──(左鍵)──> WAITING ──(咬勾提示)──> TENSION ──(進度條滿)──> FINISH
  ^                                                   |                |
  |                                          (進度條空，失敗)        (左鍵)
  └────────────────────────────────────────────────────────────────────┘
```

四個狀態：`IDLE` → `WAITING` → `TENSION` → `FINISH`

## 各階段分析

### 1. IDLE 階段（`image_idle.png`）

**特徵**
- 畫面中央下方有釣竿與捲線器，無任何 UI 浮窗
- 無「!」、無拉力計、無釣起魚的浮動文字

**偵測策略**
- 以排除法為主：其他三個狀態均有明顯視覺特徵，偵測不到任何特徵時判定為 IDLE
- 可額外偵測「浮標/餌」是否不在水中（捲線器區域顏色特徵）

**行動**
- 按下左鍵（拋竿）→ 轉入 WAITING
- 加入隨機延遲（500ms ± 100ms）避免機器行為過於規律

---

### 2. WAITING 階段（`image_waiting.png`）

**特徵**
- 釣竿垂入水中，畫面相對靜態
- 無拉力計 UI，無「!」提示

**咬勾偵測方案（建議雙重判斷）**

#### 方案 A：視覺偵測「!」像素藝術圖示
從附圖（image1~5.png）觀察：
- 「!」為像素風格白色本體＋藍紫色外框，整體呈高對比
- 大小與位置隨鏡頭角度變化（3D 遊戲），image1 展示了畫面略微朝下時「!」可能超出上方畫面邊界
- 偵測方式：
  1. **模板匹配（Template Matching）**：準備多個不同大小的「!」模板，使用 `cv2.matchTemplate` 對全螢幕滑動搜尋，設定相似度閾值（建議 0.75 以上）
  2. **顏色+形狀 Blob 偵測**：針對高亮白色區域（HSV 高 V、低 S）進行 Blob 分析，篩選細長長方形（符合「!」主體比例）
  3. **差分偵測**：兩張相隔約 200ms 的截圖做差分，「!」閃爍時差值大，可觸發判斷

#### 方案 B：音訊指紋比對（推薦為主判斷）✨ 支援參考錄音

**你可以提供咬勾音效的錄音檔（WAV/MP3），程式會以此作為比對基準，大幅提升準確率。**

**原理：滑動視窗相關性比對（Cross-Correlation）**

```
參考音效（reference.wav）
    ↓ 預處理（normalize、降噪）
    ↓ 提取特徵向量
即時音訊串流（每 ~50ms 一個 chunk）
    ↓ 與參考特徵做相似度計算
    ↓ 相似度 > 閾值 → 觸發咬勾
```

**實作方案比較**

| 方法 | 原理 | 優點 | 缺點 |
|------|------|------|------|
| **波形互相關**（`scipy.signal.correlate`） | 時域滑動相關 | 簡單快速，對精確音效效果好 | 對音量變化敏感，需正規化 |
| **MFCC 特徵比對**（`librosa`） | 頻譜梅爾倒頻係數 | 對音量/噪音更具魯棒性 | 需要 `librosa`，稍慢 |
| **頻譜能量分佈**（FFT 特徵） | 頻域能量分佈比對 | 輕量，適合即時處理 | 需人工分析音效頻率特徵 |

**建議方案：正規化波形互相關**
1. 啟動時載入 `bite_sound.wav`（使用者提供），裁切出主要音效波形並正規化
2. 持續從系統音訊（`sounddevice` 的 loopback 模式）擷取串流
3. 每個新 chunk 與參考波形做互相關，取最大相關係數
4. 相關係數 > `BITE_CORR_THRESHOLD`（建議初始值 0.6）→ 觸發
5. 觸發後設置冷卻時間（cooldown ~2 秒），避免重複觸發

**錄音建議**
- 格式：WAV（16-bit, 44100Hz 或 48000Hz）
- 內容：只錄咬勾音效本體（1 秒以內），前後靜音盡量短
- 放置路徑：`assets/bite_sound.wav`

**若無法提供錄音（備援）**：退回 RMS 閾值方案，但僅在 WAITING 狀態啟用，並對特定頻率範圍（通過 bandpass filter）做 RMS，降低環境音誤觸率。

#### 建議策略
**音訊指紋為主 + 視覺「!」為輔**，任一偵測到即觸發。音訊方案完全不受 3D 鏡頭角度影響，是最穩定的咬勾偵測手段。

**行動**
- 偵測到咬勾提示後立即按下左鍵 → 轉入 TENSION
- 若超過設定等待上限（例如 60 秒）未咬勾，重新拋竿（左鍵 → 回 IDLE/WAITING）

### 3. TENSION 階段（`image_tension.png`）

**UI 結構**（從附圖分析）

```
  ┌─────┐  ┌──────┐
  │▓▓▓▓▓│  │      │  ← 空白（魚在白區上方，力量過大）
  │▓▓▓▓▓│  │██████│  ← 白色區域（適當拉力區間，隨按鍵上下移動）
  │     │  │  🐟  │  ← 魚圖示（位置固定或緩慢移動）
  │     │  │      │  ← 空白（魚在白區下方，力量過小）
  └─────┘  └──────┘
  進度條    拉力條
 （左側）   （右側，藍色外框）
```

**進度條（左側黑色長條）**
- 從底部向上綠色填充
- 填滿頂部 → 成功
- 退到底部 → 失敗
- 偵測：掃描進度條 ROI 中綠色像素比例（HSV: H≈90-150, S>100, V>100）

**拉力條（右側藍框）**
- 白色區域：持續下墜，按壓左鍵上升
- 魚圖示：固定於某 Y 位置，或緩慢漂移（依實際情況調整）
- 圖示顏色多變（綠、橘、紅等），以形狀特徵為主（小型像素風魚形）

**控制策略**

| 狀況 | 行動 |
|------|------|
| 魚在白色區域內 | 進入適當拉力，進度條前進 → 維持當前按鍵狀態 |
| 白色區域下緣即將低於魚 | 按壓左鍵，讓白色區域上升 |
| 白色區域上緣高於魚太多 | 放開左鍵，讓白色區域下降 |

**建議演算法（簡易 PD 控制）**

1. 偵測白色區域的中心 Y（`white_center_y`）
2. 偵測魚圖示的中心 Y（`fish_y`）
3. 計算誤差 `error = fish_y - white_center_y`（正值表示魚在白區下方）
4. 若 `error > threshold_down`：放開左鍵
5. 若 `error < threshold_up`：按下左鍵
6. 加入死區（dead zone）避免頻繁切換

**ROI 定位方式（動態全畫面掃描）**

拉力計同樣受 3D 鏡頭角度影響，UI 會出現在畫面不同位置（附圖顯示夜晚場景中出現在偏下方）。

**方案：每幀動態搜尋藍色霓虹外框**

1. **顏色遮罩**：對全畫面套用 HSV 藍色範圍（H≈200-240, S>150, V>150）取得遮罩
2. **輪廓分析（`cv2.findContours`）**：在遮罩中尋找封閉矩形輪廓
3. **幾何篩選**：
   - 長寬比接近 1:4～1:6（細長直立矩形）
   - 面積在合理範圍內（排除噪點與過大區塊）
   - 同時找到兩個相鄰的細長矩形（進度條 + 拉力條並排）
4. **鎖定 UI 位置**：找到後記錄 bounding box，下一幀優先在此附近搜尋（減少計算量）；若搜尋失敗則重新全畫面掃描
5. **狀態切換依據**：若藍框消失超過 N 幀（例如 5 幀）→ TENSION 結束（成功或失敗由進度條最後狀態決定）

**額外保險：結構一致性驗證**
- 找到候選框後，驗證其內部是否同時存在：左側黑色長條（進度條特徵）+ 右側白色區塊（拉力區特徵）
- 若驗證失敗則視為誤判，繼續搜尋

**拉力計超出畫面時的處理原則**

> 捲線器永遠在畫面正下方，無法作為鏡頭調整的視覺錨點。拉力計 UI 是唯一的偵測依據，但當它被截斷時只剩部分可見，完全消失時則無任何參考資訊。因此**放棄所有主動移動鏡頭的方案**，理由如下：
> - 程式無從得知應往哪個方向移動（UI 消失 = 零資訊）
> - VRChat 滑鼠靈敏度由玩家設定，無法換算角度
> - TENSION 期間左鍵已被佔用，邏輯互相干擾風險高

**實際處理策略：接受失敗、快速重試**

| 情況 | 判斷方式 | 行動 |
|------|---------|------|
| UI 完整可見 | 藍框完整偵測到 | 正常執行拉力控制 |
| UI 被邊緣截斷（部分可見） | 藍框輪廓不完整，但仍可偵測到部分 | 嘗試從可見區域估算進度條與拉力條位置，繼續控制 |
| UI 完全消失 | 連續 N 幀（建議 10 幀 ≈ 160ms）找不到藍框 | **視為 TENSION 失敗**，直接回到 IDLE 重新開始釣魚 |

**為何「直接放棄重試」是合理選擇**
- 釣魚週期短，失去一條魚損失極低
- 盲目移動鏡頭可能造成連續失誤（找不到方向 → 一直失敗）
- 程式完全不需要感知「鏡頭角度」這個無法讀取的狀態

**行動**
- 進度條滿 → 轉入 FINISH
- 進度條空，或 UI 持續消失超過閾值 → 視為失敗，回到 IDLE

---

### 4. FINISH 階段（`image_finish.png`）

**特徵**
- 畫面出現「+XX XP」浮動文字（黃綠色）
- 出現魚名稱與重量文字（例如「BonFish (3.6kg)」）
- 可能有魚的像素圖示飄起

**偵測策略**
- **方案 A（OCR）**：使用 `pytesseract` 偵測「+」開頭的 XP 浮字，或偵測包含「kg」的文字
- **方案 B（顏色偵測）**：偵測畫面出現特定黃綠色亮字（HSV 黃綠範圍），比 OCR 更快速
- **方案 C（差分）**：從 TENSION 狀態結束後，偵測拉力計 UI 消失即可判定（進度條滿後 UI 消失 → 成功）

**行動**
- 等待約 1~2 秒（欣賞魚）
- 按下左鍵 → 回到 IDLE
- 記錄捕獲魚種與 XP（logging）

## 技術細節

### 畫面擷取
- 使用 `mss` 進行高速截圖（建議 30fps 以上輪詢速度）
- 針對各 ROI 分別擷取，減少每次處理的像素量
- 截圖解析度以 1366×768（附圖解析度）為基準，加入縮放係數以適配其他解析度

### ROI 定義（基於 1366×768，需實測微調）

```python
ROI = {
    "tension_ui":     (450, 250, 350, 350),  # (x, y, w, h) 拉力計整體
    "progress_bar":   (450, 280, 60, 280),   # 進度條
    "tension_bar":    (580, 280, 80, 280),   # 拉力條
    "finish_text":    (400, 350, 500, 200),  # 釣起浮字區域
    "exclaim_search": (0, 0, 1366, 600),     # 「!」搜尋範圍（上半畫面）
}
```

### 輸入模擬
- 使用 `win32api.mouse_event` 或 `pydirectinput` 發送滑鼠事件，避免 VRChat 鍵盤鉤子攔截
- 按下 / 放開要分開處理（不使用 click，使用 mousedown + mouseup）

### 狀態機迴圈建議頻率

| 狀態 | 輪詢間隔 | 原因 |
|------|---------|------|
| IDLE | 200ms | 無急迫性 |
| WAITING | 50ms（視覺）/ 即時（音訊） | 需快速反應咬勾 |
| TENSION | 16ms（~60fps） | 需即時控制拉力 |
| FINISH | 200ms | 無急迫性 |

## 主要風險與對策

| 風險 | 對策 |
|------|------|
| 3D 鏡頭角度導致「!」偏移出畫面（image1） | 允許部分匹配，搜尋範圍擴及畫面邊緣；音訊偵測作為主要手段 |
| 環境音誤觸音訊偵測 | 使用參考錄音做波形互相關比對，比單純 RMS 閾值精準數倍；僅在 WAITING 狀態啟用偵測 |
| 拉力計 UI 位置隨鏡頭移動（如夜晚場景偏下） | 每幀全畫面掃描藍色霓虹外框輪廓，動態定位 UI；UI 完全消失超過閾值則視為失敗直接重試，不做任何鏡頭移動 |
| 遊戲畫面非全螢幕 / 解析度不同 | 先取得 VRChat 視窗 HWND，對視窗座標系進行截圖（`win32gui` + `mss`），並加入解析度縮放 |
| 時機過早按左鍵（咬勾提示剛出現） | 加入 50~100ms 隨機延遲後再點擊，模擬人類反應 |

## 開發建議順序

1. **環境搭建**：安裝依賴、確認 VRChat 視窗截圖可行
2. **截圖模組**：實作 `screen_capture.py`，加入 debug 視覺化（圈出偵測到的 ROI）
3. **狀態偵測**（從最有特徵的開始）：FINISH → TENSION UI → BITE「!」→ IDLE/WAITING 排除
4. **音訊指紋模組**：`audio_detector.py`，載入 `assets/bite_sound.wav`，實作波形互相關偵測，校準相關係數閾值
5. **拉力計控制**：`tension_handler.py`，先以 print 輸出白區/魚位置，確認偵測正確後接入輸入
6. **完整狀態機串接**：`main.py` 整合所有模組
7. **參數調優**：實測各閾值、延遲、ROI 座標
